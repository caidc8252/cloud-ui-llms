# 颜色

颜色用于建立视觉层级、标识交互、传达状态并区分数据。@cloud/ui 调色板以语义化设计令牌（design token）的形式定义，因此同一套组件标记在浅色与深色主题下都能保持语义一致。

## 明暗模式合规

在应用代码中请使用语义化工具类（utility）或组件公开属性。浅色值为默认值；当根元素带有 `.dark` 或 `[data-theme="dark"]` 时，深色值生效。

下列表格列出了 @cloud/ui 当前声明的每个颜色令牌的解析值。别名令牌已解析为其底层对应模式的值；派生的 `color-mix()` 令牌保留为表达式，并按各模式展开其引用的令牌值。每条 **说明** 会解释该令牌的用途、给出一两个具体使用场景，并在容易误用的令牌上标注不应在何处使用。

权威来源为 the shared stylesheet。这些值仅作参考数据，并非鼓励你把 OKLCH 字面量复制进业务 UI。

| `--color-ring`                 | `oklch(40% 0.14 262)`   | `oklch(70% 0.13 262)`  | focus-visible 颜色的遗留 shadcn 别名。新代码中请优先 `line-focus`。                             |

## 用色指南

@cloud/ui 的用色之道在于克制。任何屏幕的绝大部分都应是中性的——surface、content 与 line 令牌——如此一来，你落下的少数高饱和色才会读作有意为之的信号：这是操作、这坏了、这需要注意。当每个面板都以自己的色相争夺注意力时，谁都赢不了。请把强色当作稀缺资源，只花在用户接下来最该看的那一件事上，其余一切交给中性色承载。

下面每个色彩角色都列出了它所*适用*的场景，以及同样重要的——在哪些场景下动用它是一种错误。请依角色的含义来选择令牌，绝不要凭某个色板恰好在你的组件旁看起来如何来选。

### 用中性色建立层级

中性色是默认。页面的大部分——面板、表格、表单、工具栏、空状态、正文——都应完全由 surface、content 与 line 令牌构建，完全不用任何调色板颜色。

**表面（Surfaces）** 层叠以表达深度。用 `surface-1` 作为应用画布与页面背景，即其余一切所在的平面。用 `surface-2` 作为主组件表面：卡片、面板、popover、菜单、表格主体，以及任何应从画布上抬升的内容块。用 `surface-3` 表示后退的区域——卡片内的嵌套凹槽、低调侧栏、禁用或只读区域、区块标题后的色带。仅通过元素自身的 `:hover`/按下态去动用 `surface-hover` 与 `surface-active`，并用 `surface-overlay` 作为模态框、抽屉或对话框之后使页面变暗的遮罩。不要靠降低不透明度或凭眼睛微调某个灰度档位来模拟抬升；请选用其角色与你正在构建的层级相匹配的表面令牌。

**内容（Content）** 表达阅读优先级，而非装饰。用 `content-primary` 表示用户最先阅读的文字与图标——标题、数值、正文、激活标签。用 `content-secondary` 表示伴随主层的支撑性文字——帮助文本、描述、非激活但仍可读的标签。用 `content-tertiary` 表示最安静但仍可读的一层：时间戳、元数据、计数、占位式提示、图表标注。仅对真正禁用的控件使用 `content-disabled`——它刻意对比过低，不适合任何用户必须阅读之物，因此切勿仅为显得低调而用它替代三级文字。当前景位于深色或反转表面上时用 `content-inverse`，位于实心主色操作表面上时用 `content-on-primary`；这些情形都不要手挑浅灰。

**线条（Lines）** 分隔而不喧哗。用 `line-subtle` 表示最低强调的分隔——列表行之间的分隔线、结构性图表网格线。用 `line-default` 表示控件与组件的标准边界——输入框边框、卡片边缘、表格单元格边框。仅在少数确实需要更重边界之处保留 `line-strong`，如分隔两个否则会糊在一起的密集区域。`line-focus` 由交互原语的 focus-visible 处理持有——不要把它挪作普通边框。

### 强调操作与交互

主色标记某一情境下唯一最重要的操作，以及归属于组件的交互状态。它最常见的职责，是代表屏幕上最该做之事的那个实心按钮——Save、Submit、Continue、Create——以及链接、选中的导航和交互元素上的点缀。在任一给定视图中，通常应恰好只有一个主色操作争夺眼球；其余的请以中性或描边处理呈现，使那唯一的主色操作保持明确。若两个按钮都是主色，用户就不再知道你要他们按哪一个。

让交互色始终绑定产生它的状态。`surface-hover` 与 `surface-active` 属于 `:hover` 与按下态；`state-selected` 属于被选中的行或项；`line-focus`（配合原语的聚焦环）属于 `:focus-visible`。选中必须与悬停在视觉上保持区分，且当二者可能同时生效时——某行既被悬停又被选中——选中必须胜出。行悬停浸染（`state-row-hover`）刻意弱于操作自身的悬停，使得悬停一张密集表格时，不会盖过其内部按钮的悬停反馈。

强调色是专用色，而非第二个主色。它保留给图表功能项与 AI/助手表面——AI 面板背后的强调淡色、生成式功能项上的高亮、图表相邻控件。不要用强调色让普通业务按钮、标签页或横幅“跳出来”；那是主色的职责，为此借用强调色会抹除它本应承载的信号。

### 编码状态与关联

状态色报告某个真实事物的健康或严重程度。仅当颜色在回答“它状况如何？”时才使用——不作装饰，也不作类别编码。

- **成功（绿色）** 确认某事已完成或健康：保存成功的 toast、“活跃”/“通过”/“健康”徽章、已完成的步骤、正向校验状态。用 `success-bg` 作横幅或徽章的淡色填充，`success` 作图标或指示，`success-strong` 作必须在淡色上保持清晰的文字。
- **警告（琥珀）** 标记需要留意但尚未失败之事：逼近的配额、即将过期的凭据、降级但仍运行的资源、非阻断的表单警示。它表达“继续前先看一眼”，而非“已经坏了”。
- **错误（红色）** 标记失败、破坏或阻断性问题：失败的请求横幅、无效字段、“失败”/“离线”徽章，或破坏性操作的强调。`error-active` 是危险操作的按下态（删除按钮被点击时）；请将其绑定到该按下动作，而非当作更亮的红色使用。
- **信息（蓝色）** 承载中性、非严重程度的通知：信息标注、“新功能”说明、既非好也非坏消息的提示或情境解释。

始终为状态色搭配第二线索——图标、标签或文字本身——使含义在无法看见颜色时仍能留存，也使红/绿不再是区分通过与失败的唯一凭据。类别、合同类型、标签或普通元数据值**不是**状态：请让它们保持中性（或按下文赋予类别标识），而不要为它们借用绿或红，那会错误地暗示健康或失败。

### 标识类别

类别色为某一*类*事物赋予稳定、可复用的身份——而非对它的评判。当你需要同一类别在其出现的每一处都长得一样，且类别数量小而固定时使用。

`cat-1` … `cat-6` 成对色是组件自有的标识处理（例如 ColorTile），为类别分配固定色相。始终让前景与其对应背景成对——`cat-2` 填充配 `cat-2-fg` 文字——以使对比在明暗两种模式下都正确，并用 `cat-line` 作方块与标签周围的共享细线。装饰性青色与紫罗兰色相同样充当额外的类别槽位；与 `cat-*` 成对色一样，它们是类别色，**绝不**可替代状态。青色徽章不应读作“成功”，紫罗兰徽章不应读作“警告”——若某个值意味着健康或严重程度，请改用状态角色。

### 可视化数据

图表拥有自己的调色板，使数据色与应用框架保持分离。不要把图表色引入按钮、徽章或背景，也不要把主色/状态色引入图表去表达它们本无之意。

- **分类**（`chart-1` … `chart-8`）：按数字顺序、自 `chart-1` 起分配给离散系列，使同一第 N 个系列在每张图表中获得相同颜色。该调色板已排序，以获得相邻系列间的最大可区分度。
- **顺序**（`chart-seq-100` … `chart-seq-700`）：用于单一有序量级——热力图、密度、由低到高的渐变——越浅表示越少，越深表示越多。
- **发散**（`chart-div-neg` / `chart-div-mid` / `chart-div-pos`）：当值自负、经有意义的零、到正变化时使用，使中性中点标记交叉处，两侧各自读作其方向。

用 `chart-grid`、`chart-axis`、`chart-label`、`chart-dim` 以及 `chart-tooltip-*`/`chart-zoom-*` 令牌构建图表结构，使网格线与坐标轴比数据更安静。当某个有文档的图表或组件 API 需要时，允许作用域内的语义 CSS 变量引用——请优先采用该组件契约，而非复制一个已解析的 OKLCH 值。

### 承载品牌标识与头像

单色品牌令牌渲染出可随模式适配的单色品牌标识：`brand-mono` 为默认权重，`brand-mono-strong` 用于标识需要最高对比之处（繁忙表面上的报头），`brand-mono-soft` 用于更安静、类似水印的摆放。头像默认以 `avatar-bg`/`avatar-fg` 作其背景与首字母或回退图标。它们属于框架，而非状态或类别色——不要为传达状态而重新着色。

## 无障碍

颜色绝不能是传达信息的唯一方式。请补充文字、图标、标签或组件状态，使含义在颜色无法被感知时仍然可得。

在前景与其目标背景之间保持足够对比。在原语提供处保留组件自有的 focus-visible 处理，包括 `border-line-focus` 与 `shadow-focus`。不要用三级内容色替代专用的禁用色。

## 实现

通过生成的工具类（如 `bg-surface-2`、`text-content-primary`、`border-line-default`）或组件公开属性（如 `tone` 与 `variant`）来消费令牌名。

不要在业务应用代码中引入裸 hex 或 OKLCH 值、任意颜色工具类，或页面局部颜色变量。直接的语义变量引用仅限于需要作用域 CSS 值的有文档 API，包括图表组合。
