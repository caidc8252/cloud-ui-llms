# Colors

[中文](./colors.zh-CN.md)

Colors establish visual hierarchy, identify interaction, communicate status, and distinguish data. The @cloud/ui palette is defined as semantic design tokens so the same component markup remains meaningful in light and dark themes.

## Visual mode compliance

Use semantic utilities or public component props in application code. Light values are the default. A root with `.dark` or `[data-theme="dark"]` activates the dark values.

The tables list the resolved value for every color token currently declared by @cloud/ui. Alias tokens are resolved to their underlying mode-specific value. Derived `color-mix()` tokens remain expressions, with their referenced token values expanded for each mode. Each **Description** explains what the token is for, gives a concrete example or two of where it appears, and — where the token is easy to misuse — notes where not to reach for it.

The canonical source is the shared stylesheet. These values are reference data, not an invitation to copy OKLCH literals into business UI.

## Using color

Color in @cloud/ui works by restraint. The large majority of any screen is neutral — surface, content, and line tokens — so that the few saturated colors you do place read as deliberate signals: this is the action, this is broken, this needs attention. When every panel competes for attention with its own hue, none of them win. Treat strong color as a scarce resource you spend on the one thing the user should look at next, and let neutrals carry everything else.

Each color role below lists the situations it is _for_ and, just as importantly, the situations where reaching for it is a mistake. Choose a token by the meaning of the role, never by how a swatch happens to look next to your component.

### Build hierarchy with neutral colors

Neutrals are the default. Most of a page — panels, tables, forms, toolbars, empty states, body copy — should be built entirely from surface, content, and line tokens, with no palette color at all.

**Surfaces** stack to express depth. Use `surface-1` for the application canvas and page background, the plane everything else sits on. Use `surface-2` for the primary component surface: cards, panels, popovers, menus, table bodies, and any content block that should feel lifted off the canvas. Use `surface-3` for regions that recede — a nested well inside a card, a muted sidebar, a disabled or read-only region, the striped band behind a section header. Reach for `surface-hover` and `surface-active` only through an element's own `:hover`/pressed states, and use `surface-overlay` for the scrim that dims the page behind a modal, drawer, or dialog. Do not simulate elevation by lowering opacity or nudging a gray palette step by eye; pick the surface token whose role matches the layer you are building.

**Content** expresses reading priority, not decoration. Use `content-primary` for the text and icons a user reads first — headings, values, body copy, active labels. Use `content-secondary` for supporting text that accompanies the primary — helper text, descriptions, inactive-but-legible labels. Use `content-tertiary` for the quietest legible layer: timestamps, metadata, counts, placeholder-style hints, and chart annotations. Use `content-disabled` only for genuinely disabled controls — it is intentionally too low-contrast for anything a user must read, so never substitute it for tertiary text merely to look subtle. When a foreground sits on a dark or inverted surface use `content-inverse`, and when it sits on a filled primary action surface use `content-on-primary`; do not hand-pick a light gray for those cases.

**Lines** separate without shouting. Use `line-subtle` for the lowest-emphasis separators — divider rules between list rows, structural chart gridlines. Use `line-default` for the standard boundary of controls and components — input borders, card edges, table cell borders. Reserve `line-strong` for the few places that genuinely need a heavier boundary, such as separating two dense regions that would otherwise blur together. `line-focus` is owned by the interactive primitive's focus-visible treatment — do not repurpose it as an ordinary border.

### Highlight actions and interaction

Primary color marks the single most important action in a given context and the interactive states that belong to components. Its most common job is the one filled button that represents the main thing to do on a screen — Save, Submit, Continue, Create — plus links, selected navigation, and the accent on interactive elements. On any given view there should usually be exactly one primary action competing for the eye; render the rest as neutral or outline treatments so the primary one stays unambiguous. If two buttons are both primary-colored, the user no longer knows which one you want them to press.

Keep interaction colors attached to the state that produces them. `surface-hover` and `surface-active` belong to `:hover` and pressed states; `state-selected` belongs to a chosen row or item; `line-focus` (with the primitive's focus ring) belongs to `:focus-visible`. Selected must stay visually distinct from hover, and where both can apply at once — a hovered row that is also selected — selected has to win. The row-hover wash (`state-row-hover`) is deliberately weaker than an action's own hover so that hovering a dense table doesn't overpower the hover feedback on the buttons inside it.

Accent color is a specialist, not a second primary. It is reserved for chart affordances and AI/assistant surfaces — the accent tint behind an AI panel, the highlight on a generative affordance, a chart-adjacent control. Do not use accent to make an ordinary business button, tab, or banner "pop"; that job belongs to primary, and borrowing accent for it erases the signal that accent is supposed to carry.

### Encode status and associations

Status colors report on the health or severity of something real. Use them only when the color is answering "how is this doing?" — not as decoration and not as category coding.

- **Success (green)** confirms that something completed or is healthy: a saved-successfully toast, an "Active"/"Passing"/"Healthy" badge, a completed step, a positive validation state. Use `success-bg` for the tinted banner or badge fill, `success` for the icon or indicator, and `success-strong` for text that must stay legible on the tint.
- **Warning (amber)** flags something that needs attention but is not yet a failure: an approaching quota, a soon-to-expire credential, a degraded-but-running resource, a non-blocking form caution. It says "look before you continue," not "this is broken."
- **Error (red)** marks failure, destruction, or blocking problems: a failed request banner, an invalid field, a "Failed"/"Offline" badge, or the emphasis on a destructive action. `error-active` is the pressed state for a danger action (a Delete button being clicked); keep it wired to that press, not used as a brighter red.
- **Info (blue)** carries neutral, non-severity notices: an informational callout, a "what's new" note, a tip or contextual explanation that isn't good or bad news.

Always pair a status color with a second cue — an icon, a label, or the words themselves — so the meaning survives when color can't be seen and so red/green aren't the only thing separating pass from fail. A category, contract type, tag, or ordinary metadata value is **not** status: leave it neutral (or give it category identity, below) rather than borrowing green or red for it, which would falsely imply health or failure.

### Identify categories

Category color gives a stable, repeatable identity to a _kind_ of thing — not a judgment about it. Use it when you need the same category to look the same everywhere it appears, and when the number of categories is small and fixed.

The `cat-1` … `cat-6` pairs are component-owned identity treatments (for example ColorTile) that assign a fixed hue to a category. Always keep a foreground with its matching background — `cat-2` fill with `cat-2-fg` text — so contrast stays correct in both modes, and use `cat-line` for the shared hairline around tiles and labels. The decorative teal and violet hues serve the same purpose as extra category slots; like the `cat-*` pairs, they are category colors and must **never** stand in for status. A teal badge should not read as "success," and a violet one should not read as "warning" — if a value means health or severity, use the status roles instead.

### Visualize data

Charts have their own palettes so that data color stays separate from application chrome. Do not pull chart colors into buttons, badges, or backgrounds, and do not pull primary/status colors into a chart to mean something they don't.

- **Categorical** (`chart-1` … `chart-8`): assign to discrete series in numeric order, starting at `chart-1`, so the same Nth series gets the same color across every chart. The palette is ordered for maximum separability between adjacent series.
- **Sequential** (`chart-seq-100` … `chart-seq-700`): use for a single ordered magnitude — heatmaps, density, a low-to-high ramp — where lighter means less and darker means more.
- **Diverging** (`chart-div-neg` / `chart-div-mid` / `chart-div-pos`): use when values run from negative through a meaningful zero to positive, so the neutral midpoint marks the crossover and each side reads as its own direction.

Use `chart-grid`, `chart-axis`, `chart-label`, `chart-dim`, and the `chart-tooltip-*`/`chart-zoom-*` tokens for chart structure so gridlines and axes stay quieter than the data. Scoped semantic CSS-variable references are permitted where a documented chart or component API requires them — prefer that component contract over duplicating a resolved OKLCH value.

### Carry the brand mark and avatars

The monochrome brand tokens render a single-color brand mark that adapts across modes: `brand-mono` for the default weight, `brand-mono-strong` where the mark needs the highest contrast (a masthead on a busy surface), and `brand-mono-soft` for a quieter, watermark-like placement. Avatars default to `avatar-bg`/`avatar-fg` for their background and initials or fallback icon. These are chrome, not status or category color — do not re-tint them to signal state.

## Accessibility

Color must not be the only way information is communicated. Add text, an icon, a label, or a component state so the meaning remains available when color cannot be perceived.

Maintain sufficient contrast between foregrounds and their intended backgrounds. Preserve component-owned focus-visible treatment, including `border-line-focus` and `shadow-focus` where the primitive provides them. Do not use tertiary content in place of the dedicated disabled color.

## Implementation

Consume token names through generated utilities such as `bg-surface-2`, `text-content-primary`, and `border-line-default`, or through public component props such as `tone` and `variant`.

Do not introduce raw hex or OKLCH values, arbitrary color utilities, or page-local color variables in business application code. Direct semantic variable references are limited to documented APIs that require scoped CSS values, including chart composition.
